tech used during last year

November 8, 2014

As you will see in a bit, I’ve been quite busy during the last year. This and the lack of discipline in regards to my blog, was the reason why I haven’t written anything here in a while. But during my first year working for ThoughtWorks, I had the opportunity to work with totally differen tech stacks, and I thought, it is probably not the worst idea, to wirte down some of my experiences. I hope, you will also find it relevant/interesting.

Clever code can hurt#

  • compile time -> pain in regards to a quick TDD feedback loop
  • implicit all over the code -> team fluctuation, results in an unknown magic
  • traits -> recompaling all the codebase as soon as you change only one of them
  • different styles all over the code base
  • explosion of functional tests -> overuse of traits (e.g. single test with 17 different traits) and syntactic sugar
  • templates full of logic (example?)
  • functional programming !
  • discovering the power of the idomatic style & necessity of similiar style in the code base
  • expressivnes
  • XML literals
  • scalatra & jade

Going fast#

  • quick TDD feedback loop: jasmine with karma runner

  • nice functional collection processing with lodash

  • functions as first class citizens

  • very quick, visible progress

  • angular.js assuring structure

  • decoupling the business logic, no dependencies to angular -> even nicer tests

  • JavaScript, the good parts FTW!

  • light backend: JDK8, spring mvc for endpoint definition (nashorn for some code sharing between client and server), mongodb

  • ansible for provisioning, started very early, a good investment

  • hard to model the domain, as one is used to (typical OO inharitence)

Bleeding edge#

  • Swift is a much, much nicer languar then ObjC
  • offering functions as first class citizens and also objects, it’s acutally very similar to Scala -> easy start

  • lacking of namespacing hurts when the codebase is getting bigger and bigger, as the team needs to come up with naming conventions and all that stuff which we gladly forgott as we discovered DDD

  • XCTest is a very rudimentary testing framework

  • don’t get me started ranting about XCode -> even in the 6.1 version, no refactoring support, frequent crashes and wrong compiler erros!

  • compile time can potentially be a problem when the codebase get’s bigger (3rd month in development we already needed to adress that)

  • iOS development feels very diffrent, to what I’ve been used to coming from the jvm
